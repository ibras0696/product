# Граф «История Чечни» (узлы + рёбра, JSON)

Связный граф людей, мест и событий Чечни, собранный из исследовательской БД
[../sqlite.db](../sqlite.db). Типы узлов и рёбер совпадают со значениями
backend-каталога (`EntityType` / `RelationType`), поэтому граф ложится на
`catalog_entities` / `catalog_relations` без переименований.

Главная ценность — **связи**: герой привязан к своему селу (`born_in`), к местам
жизни/работы (`lived_in`, `worked_in`), к событиям (`participated_in`), а
события — к местам (`located_in`).

## Как собрать

```bash
python3 docs/test/build_db.py            # сначала пересобрать sqlite.db
python3 docs/test/graph/build_graph.py   # затем граф -> docs/test/graph/
python3 -m pytest docs/test/graph/test_graph.py -q
```

Генератор идемпотентен и **не меняет исходные файлы** `docs/test/*`: он только
читает `sqlite.db` и перезаписывает содержимое `graph/`. Зависимости — только
стандартная библиотека Python 3.

## Структура папки

| Путь | Роль |
| --- | --- |
| `nodes/settlements.json` | узлы-поселения (город/село/локация) |
| `nodes/people.json` | узлы-личности |
| `nodes/events.json` | узлы-события |
| `nodes/landmarks.json` | башни, крепости, мемориалы |
| `nodes/cultural_objects.json` | мечети, музеи |
| `nodes/natural_objects.json` | реки, горы, озёра |
| `nodes/discoveries.json` | открытия и изобретения (`type: artifact`) |
| `categories/*.json` | те же узлы, разложенные по 5 категориям интерфейса (см. ниже) |
| `chronology.json` | события по возрастанию даты: точная дата, описание, тип, место, участники и источники |
| `edges.json` | все рёбра графа |
| `sources.json` | справочник использованных источников |
| `curated_discoveries.json` | **ручной** источник открытий/изобретений (см. ниже) |
| `curated_people.json` | **ручное** обогащение биографий и современные личности |
| `curated_places.json` | **ручное** обогащение описаний городов и мечетей |
| `curated_events.json` | **ручной** источник новых исторических событий и обогащения существующих |
| `report.json` | отчёт сборки: отклонённые, дедуп, обогащённые, low-confidence и т.д. |
| `build_graph.py` | генератор |
| `graph_schema.py` | словарь типов и очистка текста |
| `graph_dedup.py` | дедупликация узлов и рёбер |
| `graph_output.py` | запись категорий и хронологии |
| `test_graph.py` | проверки целостности графа |

Все файлы `curated_*.json` рядом с генератором подхватываются автоматически.

Узлы разложены по типам в отдельные файлы — так удобно дозаписывать вручную:
добавляешь объект в нужный `nodes/*.json` и связь в `edges.json`.

## Формат узла

```json
{
  "slug": "visaitov",
  "type": "person",
  "name_ru": "Мовлид Алероевич Висаитов",
  "name_ce": null,
  "aliases": [],
  "description": "Чистый текст для пользователя — без ID, URL и статусов.",
  "external_id": "Q4112141",
  "provenance": { "record_url": "https://…" },
  "sources": ["warheroes"],
  "status": "verified"
}
```

Правила текста и полей:

- `description` — **только человекочитаемый текст**. Технические данные
  (`external_id`, `coordinate*`, `provenance`, `status`) живут в отдельных полях,
  их нельзя вклеивать в описание.
- одинаковые ссылки не дублируются: если URL записи и URL координат совпадают, в
  `provenance` остаётся один `record_url`.
- у мест дополнительно: `place_type`, `district`, `coordinate {lat,lon}`,
  `coordinate_accuracy`, `coordinate_precision_m`, при наличии — независимое
  `provenance.osm_evidence` с расстоянием до основной точки.
- у людей дополнительно: `category`, `title`, `period_from/period_to` (годы
  жизни), `birth_date`, `death_date`.
- `photo` (если есть) — ссылка на изображение Wikimedia Commons с лицензией:
  `{ "url", "source_page", "license", "license_url", "author" }`. У людей, мест и
  событий фото подтягивается из `media_assets` БД; для курируемых записей его
  можно задать в `curated_*.json`.

## Формат ребра

```json
{
  "type": "born_in",
  "from": "visaitov",
  "to": "nadterechnaya",
  "title_ru": "Родился в",
  "evidence": { "method": "birthplace_id", "source": "birthplace_id",
                "osm_distance_km": 0.19 },
  "confidence": "high",
  "sources": ["warheroes"],
  "status": "verified"
}
```

Направление `from → to`. Типы: `born_in`, `lived_in`, `worked_in`,
`participated_in`, `located_in`, `connected_with` (для «умер/похоронен/увековечен
в»). Поле `confidence` (`high|medium|low`) отражает надёжность именно связи.

## Правило привязки «герой → село» (инфо + гео)

1. Место рождения берётся по приоритету: ручной `birthplace_id` → сматченный
   Wikidata-QID → упоминание в биографии.
2. При наличии координат добавляется гео-подтверждение: расстояние до
   независимой точки OpenStreetMap (`osm_distance_km`).
3. `confidence`:
   - `high` — точная привязка + координаты в Чечне;
   - `medium` — привязка по имени/тексту без гео-подтверждения;
   - `low` — извлечено из текста биографии, требует проверки.
4. Неоднозначные случаи (в биографии несколько сёл) не привязываются, а
   попадают в `report.json → ambiguous_birthplace`.

## Фильтр «только Чечня»

Внешние места из чужих биографий (Шлиссельбург, Караганда, Брест) и посторонние
Wikidata-строки без единого признака связи с Чечнёй в граф **не попадают** — они
перечислены в `report.json → rejected_non_chechnya`, а не удаляются молча.

Признак «относится к Чечне»: чеченский район, координаты внутри региона,
топоним-признак в названии/тексте, связь ребром с уже принятой чеченской
сущностью, либо принадлежность к курируемому корпусу `dataset.py`.

Дополнительно из «Личностей» исключаются **русские (славянские) личности**
массового слоя — определяются по русскому личному имени (см. `RUSSIAN_FIRST_NAMES`).
Курируемых чеченских героев фильтр не трогает; всё убранное перечислено в
`report.json → rejected_russian_person`, поэтому спорную запись легко вернуть.

## Категории интерфейса — папка `categories/`

Те же узлы, разложенные по **пяти категориям легенды карты** (как на фронте).
Раскладка повторяет функцию `entityKind()` бэкенда — девять `EntityType`
сводятся в пять категорий:

| Файл | Категория | Какие `type` внутри |
| --- | --- | --- |
| `naselennye-punkty.json` | Населённые пункты | `settlement`, `natural_object` |
| `lichnosti.json` | Личности | `person` |
| `sobytiya.json` | События | `event` |
| `dostoprimechatelnosti.json` | Достопримечательности | `landmark`, `cultural_object` |
| `istochniki.json` | Источники | `artifact` (открытия/изобретения) |

Информация не теряется: карточка переносится целиком, а её точный `type`
сохраняется — внутри «Населённых пунктов» видно, где село, а где река.

> Примечание: по схеме бэка `artifact` → категория `source`, поэтому открытия и
> изобретения попадают в «Источники». Библиографические источники (книги, архивы)
> — это отдельный реестр [sources.json](sources.json), а не карточки на карте.

## Открытия и изобретения (курируемый overlay)

Данные об открытиях и изобретениях **отсутствуют в `sqlite.db`** — они собраны из
веб-источников вручную и лежат в [curated_discoveries.json](curated_discoveries.json).
Генератор подмешивает их так же, как OSM-свидетельства: файл — источник истины,
`nodes/discoveries.json` и рёбра `created_by` — производные. Перезапуск сборки не
теряет эти данные.

Каждое открытие/изобретение — узел `type: artifact`, привязанный к автору ребром
`created_by → person`. Приоритет — **качество и проверяемость** (BR-008), а не
количество: включаются только именованные, подтверждённые источником достижения.

Чтобы добавить новое открытие/изобретение, допиши в `curated_discoveries.json`:

```json
{
  "artifacts": [{
    "slug": "invention-…",
    "name_ru": "…",
    "description": "Чистый текст без ID/URL/статусов.",
    "field": "Область науки",
    "kind": "Открытие | Изобретения",
    "sources": ["ru_wikipedia"], "status": "needs_review"
  }],
  "relations": [{
    "type": "created_by", "from": "invention-…", "to": "<slug автора>",
    "title_ru": "Автор", "confidence": "high",
    "sources": ["ru_wikipedia"], "status": "needs_review"
  }]
}
```

`to` — это `slug` личности: либо уже существующий узел графа (например
Wikidata-`wd-q…`), либо новый человек, добавленный в секцию `people` того же файла.
Источники из overlay объединяются с `sources.json` автоматически.

## Обогащение биографий (overrides)

Узлы из `sqlite.db` (герои, изобретатели, современные деятели) можно дополнить
готовым текстом, не трогая БД. В `curated_people.json` секция `overrides` патчит
поля существующего узла **по slug** — описание, годы, звание, источники:

```json
{ "overrides": [{
    "slug": "kadyrov-ramzan",
    "description": "Чистый биографический текст…",
    "title": "Глава Чеченской Республики, Герой России",
    "sources": ["parlament_chr", "checheninfo"]
}]}
```

Био Кадыровых, изобретателей (Цебиев, Хаджиев) и Героев Советского Союза
(Нурадилов, Идрисов, Бейбулатов и др.) собраны из чеченских и профильных
источников. Обогащённые узлы перечислены в `report.json → enriched_nodes`.

## Дедупликация — в момент записи

Исходные файлы не чистятся. При сборке:

- **узлы** сводятся по `external_id`, иначе по паре `(type, нормализованное имя)`;
  дубликат отбрасывается, его источники сливаются в оставшийся узел;
- одноимённые узлы с **почти совпадающими координатами (< 300 м)** тоже
  склеиваются; но одноимённые **далёкие** места (в Чечне есть разные сёла с одним
  названием, напр. Аллерой в 15 км друг от друга) остаются раздельными — это не
  дубликаты;
- **рёбра** сводятся по `(type, from, to)`; остаётся связь с наибольшим
  `confidence`, источники объединяются;
- всё слитое фиксируется в `report.json → deduplicated_nodes`.
