# Техническое задание

## Интерактивный цифровой музейный экспонат «Паутина истории Чечни»

**Тип продукта:** интерактивное веб-приложение для сенсорного музейного экрана
**Формат:** карта Чеченской Республики + граф исторических связей
**Назначение:** музей Чеченского государственного университета им. А. А. Кадырова
**Версия документа:** 1.0
**Статус:** ТЗ на MVP с архитектурой дальнейшего развития

---

# 1. Краткое описание проекта

«Паутина истории Чечни» — интерактивный цифровой экспонат, позволяющий посетителю исследовать историю Чеченской Республики через связи между:

* населёнными пунктами;
* историческими личностями;
* учёными;
* героями;
* событиями;
* достопримечательностями;
* природными объектами;
* культурными объектами;
* музейными материалами;
* архивными источниками.

Основной интерфейс представляет собой стилизованную карту Чеченской Республики.

Посетитель выбирает населённый пункт, после чего вокруг него раскрывается интерактивная сеть связанных объектов.

Пример:

```text
Ножай-Юрт
├── историческая личность
├── учёный
├── памятное место
├── башенный комплекс
├── историческое событие
└── архивный источник
```

Каждый открытый объект может иметь дальнейшие связи:

```text
Ножай-Юрт
    ↓
Историческая личность
    ↓
Событие
    ↓
Грозный
    ↓
Чеченский государственный университет
```

Экспонат должен постоянно расширяться. Пользователи и сотрудники музея смогут предлагать новые материалы, которые публикуются только после проверки модератором.

---

# 2. Основная идея

Посетитель не читает заранее подготовленную длинную статью.

Он самостоятельно:

1. находит интересующее место;
2. нажимает на него;
3. раскрывает связанные объекты;
4. переходит от села к человеку;
5. от человека — к событию;
6. от события — к другому месту;
7. открывает первоисточник;
8. самостоятельно строит маршрут через историю.

Таким образом история представляется не как список дат, а как живая сеть людей, мест и событий.

# 3. Проблема

Историческая информация о Чеченской Республике распределена между:

* музейными фондами;
* государственными архивами;
* книгами;
* научными публикациями;
* университетскими материалами;
* семейными архивами;
* фотографиями;
* воспоминаниями жителей;
* локальными краеведческими источниками.

Посетителю сложно увидеть связи между отдельными материалами.

Обычная карточка сообщает:

> человек родился в определённом селе и участвовал в определённых событиях.

«Паутина истории Чечни» визуально показывает:

```text
человек → село → событие → документ → другое место → университет
```

# 4. Цель проекта

Создать работающий цифровой музейный экспонат, который:

* показывает исторические связи в интерактивной форме;
* вовлекает посетителя в самостоятельное исследование;
* позволяет найти собственное село или район;
* связывает историю Чечни с историей ЧГУ;
* предоставляет проверенные источники;
* позволяет расширять базу данных;
* принимает новые материалы через модерируемые заявки;
* может работать в музее без постоянного доступа к интернету.

# 6. Привязка к музею

Экспонат должен быть встроен в общую экспозицию музея ЧГУ и связывать две основные темы:

1. историю Чеченской Республики;
2. историю Чеченского государственного университета.

Пример связи:

```text
Населённый пункт
    ↓
Известный житель
    ↓
Научная или общественная деятельность
    ↓
ЧГУ
    ↓
Факультет или научное направление
```

Экспонат не должен восприниматься как обычная справочная карта. Его основная ценность — раскрытие взаимосвязей.

# 7. Форматы взаимодействия

В приложении предусматриваются два основных режима.

## 7.1. Свободное исследование

Посетитель самостоятельно выбирает место, человека или событие.

Сценарий:

```text
Открыть карту
→ найти своё село
→ открыть связанные объекты
→ выбрать человека
→ перейти к событию
→ посмотреть источник
```

## 7.2. «Найди связь»

Игровой музейный режим.

Система предлагает посетителю найти историческую связь между двумя объектами.

Примеры:

* найти связь между селом и ЧГУ;
* найти связь между исторической личностью и определённым районом;
* найти путь от Грозного до определённого культурного объекта;
* найти человека, связанного сразу с двумя населёнными пунктами;
* определить, какие события связывают два района.

Посетитель последовательно открывает узлы, пока не найдёт необходимый путь.

Пример:

```text
Ножай-Юрт
→ историческая личность
→ образовательное событие
→ Грозный
→ ЧГУ
```

После прохождения система подсвечивает весь найденный путь.

# 8. Пользовательский сценарий в музее

## 8.1. Начальный экран

На экране отображаются:

* название «Паутина истории Чечни»;
* стилизованная карта республики;
* анимированные линии между несколькими объектами;
* строка поиска;
* кнопка «Найти своё место»;
* кнопка «Начать исследование»;
* кнопка «Найди связь».

Текстовая инструкция должна занимать не более двух коротких предложений:

> Выберите место на карте.
> Открывайте людей, события и достопримечательности, связанные с ним.

## 8.2. Выбор населённого пункта

Посетитель может:

* нажать на точку;
* воспользоваться поиском;
* выбрать район;
* выбрать населённый пункт из предложенного списка;
* приблизить карту пальцами.

## 8.3. Раскрытие паутины

После нажатия на населённый пункт:

1. карта слегка затемняется;
2. выбранный объект остаётся в центре;
3. вокруг него появляются связанные узлы;
4. связи плавно прорисовываются;
5. узлы группируются по категориям.

Цвета категорий:

| Категория             | Цвет       |
| --------------------- | ---------- |
| Населённые пункты     | зелёный    |
| Личности              | синий      |
| События               | оранжевый  |
| Достопримечательности | фиолетовый |
| Источники             | жёлтый     |
| Организации и ЧГУ     | бирюзовый  |

Цвета должны быть настраиваемыми и сопровождаться иконками, чтобы интерфейс не зависел только от цветового восприятия.

## 8.4. Открытие объекта

При выборе узла открывается боковая карточка.

Карточка содержит:

* название;
* тип объекта;
* изображение;
* краткое описание до 300 символов;
* временной период;
* место на карте;
* количество связей;
* кнопки перехода по связанным объектам;
* список подтверждающих источников;
* QR-код для открытия полной версии на телефоне.

## 8.5. Завершение сеанса

При бездействии в течение 30 секунд система показывает предупреждение:

> Продолжить исследование?

Если реакции нет ещё 10 секунд, приложение возвращается на начальный экран.

# 9. Основные функциональные требования

## 9.1. Интерактивная карта

Система должна:

* отображать территорию Чеченской Республики;
* поддерживать масштабирование;
* поддерживать перемещение карты;
* поддерживать сенсорное управление;
* показывать населённые пункты;
* показывать кластеры при отдалении;
* раскрывать отдельные объекты при приближении;
* подсвечивать выбранный объект;
* отображать линии исторических связей;
* фильтровать точки по категориям;
* ограничивать начальный вид границами республики.

Для карты рекомендуется использовать MapLibre GL JS. Он рендерит интерактивные карты из векторных тайлов непосредственно в браузере и позволяет управлять стилем карты через отдельную спецификацию. Это даёт возможность сделать собственную музейную визуальную тему, а не оставлять стандартный дизайн Google или Яндекс Карт.

## 9.2. Режим графа

Система должна отображать:

* центральный выбранный объект;
* ближайшие связи;
* тип каждой связи;
* направление связи, если оно важно;
* возможность раскрытия следующего уровня;
* возможность сворачивания узлов;
* подсветку выбранного пути;
* анимацию появления связей.

Для визуализации графа рекомендуется Cytoscape.js — библиотека для моделирования, анализа и отображения графовых структур.

## 9.3. Ограничение количества узлов

Нельзя одновременно выводить весь граф.

На одном экране должно отображаться:

* до 20 узлов для сенсорного киоска;
* максимум 40 узлов при ручном раскрытии;
* один или два уровня связей.

При большем количестве система должна показывать агрегированный узел:

```text
Ещё 17 связанных личностей
```

Это необходимо для сохранения понятного интерфейса и стабильной производительности.

## 9.4. Поиск

Поиск должен работать по:

* официальному названию;
* альтернативному названию;
* названию на русском языке;
* названию на чеченском языке;
* историческому названию;
* имени личности;
* названию достопримечательности;
* ключевым словам;
* району.

Поиск должен учитывать:

* опечатки;
* разные формы написания;
* сокращения;
* альтернативную транслитерацию.

## 9.5. Фильтры

Пользователь может включать и отключать:

* населённые пункты;
* исторические личности;
* героев;
* учёных;
* события;
* достопримечательности;
* культурные объекты;
* природные объекты;
* организации;
* источники;
* материалы, связанные с ЧГУ.

Дополнительные фильтры:

* временной период;
* район;
* подтверждённость информации;
* тип источника.

## 9.6. Хронологический фильтр

Пользователь может выбрать временной диапазон.

При изменении диапазона:

* неподходящие события скрываются;
* линии связей перестраиваются;
* карта показывает только актуальные для периода объекты.

Для MVP достаточно крупных периодов, а не точного календаря.

Пример:

* до XIX века;
* XIX век;
* начало XX века;
* советский период;
* современный период.

## 9.7. Источники

Каждый опубликованный исторический факт должен иметь источник.

Поддерживаемые виды источников:

* архивный документ;
* музейный материал;
* книга;
* научная статья;
* официальное издание;
* фотография;
* видеоматериал;
* аудиозапись;
* устное свидетельство;
* официальный интернет-ресурс.

Источник должен содержать:

* название;
* тип;
* автора;
* год;
* издателя;
* архивный шифр, если имеется;
* URL, если имеется;
* файл или изображение;
* дату добавления;
* статус проверки;
* комментарий модератора.

Устное свидетельство должно явно обозначаться как свидетельство, а не как окончательно установленный факт.

# 11. Модерация

## 11.1. Статусы заявки

```text
draft
pending
in_review
needs_revision
approved
rejected
published
archived
```

Расшифровка:

| Статус           | Значение                      |
| ---------------- | ----------------------------- |
| `draft`          | Черновик                      |
| `pending`        | Отправлена                    |
| `in_review`      | Проверяется модератором       |
| `needs_revision` | Нужны дополнительные сведения |
| `approved`       | Одобрена                      |
| `rejected`       | Отклонена                     |
| `published`      | Опубликована                  |
| `archived`       | Перенесена в архив            |

## 11.2. Процесс модерации

```text
Пользователь отправляет заявку
        ↓
Автоматическая проверка обязательных полей
        ↓
Проверка файла и формата
        ↓
Поиск возможных дубликатов
        ↓
Проверка модератором
        ↓
Проверка источника
        ↓
Создание или обновление объекта
        ↓
Публикация
```

## 11.3. Решения модератора

Модератор может:

* одобрить заявку;
* отклонить;
* запросить уточнение;
* объединить с существующей записью;
* создать новый объект;
* создать новую связь;
* изменить тип объекта;
* добавить внутренний комментарий;
* отправить материал на проверку историку.

## 11.4. История изменений

Система должна хранить:

* кто внёс изменение;
* когда;
* какие поля изменены;
* старое значение;
* новое значение;
* основание изменения;
* использованный источник.

Удаление опубликованной информации должно быть логическим, а не физическим.

# 16. Использование `.db` файла

## 16.1. MVP

Для хакатонного MVP допускается SQLite:

```text
data/history.db
```

В базе хранятся:

* стартовые объекты;
* связи;
* источники;
* заявки;
* результаты модерации.

Преимущества:

* один файл;
* простой запуск;
* не требуется отдельный сервер базы;
* удобно передавать демонстрационную сборку;
* легко сделать резервную копию.

## 16.2. Ограничения SQLite

SQLite не рекомендуется как основная production-база при одновременной работе:

* публичных пользователей;
* нескольких модераторов;
* фоновой обработки;
* большого количества записей;
* нескольких экземпляров backend.

SQLite сериализует конкурирующие операции записи и подходит для прототипа или одного музейного компьютера, но не для крупной платформы.

## 16.3. Рекомендуемый переход

```text
MVP: SQLite
       ↓
Пилот: PostgreSQL
       ↓
Музейная версия: PostgreSQL + PostGIS
```

Модели должны быть реализованы через SQLAlchemy и Alembic, чтобы миграция не требовала переписывания бизнес-логики.

# 21. Карта и картографические данные

## 21.1. Рекомендуемое решение

```text
MapLibre GL JS
+
векторные тайлы на основе OpenStreetMap
+
собственный музейный стиль
```

## 21.2. Почему не Google Maps

Google Maps можно использовать, но:

* степень изменения базового дизайна ограничена правилами платформы;
* приложение зависит от API-ключа;
* присутствуют лимиты и биллинг;
* сложнее организовать полностью автономную музейную работу;
* нельзя свободно переносить картографические материалы между поставщиками.

## 21.3. Почему не Яндекс Карты как основа

Яндекс Карты подходят для быстрого прототипа, но создают аналогичную зависимость:

* API-ключ;
* внешнее подключение;
* условия использования;
* ограниченный контроль базовой карты;
* зависимость от доступности сервиса.

## 21.4. OpenStreetMap

OpenStreetMap предоставляет картографические данные, но публичные серверы тайлов нельзя рассматривать как бесплатный production-CDN для музейной системы. Их использование регулируется отдельной политикой, поэтому для внедрения нужен коммерческий поставщик тайлов либо локальный сервер.

## 21.5. Работа без интернета

Музейная версия должна иметь:

* локальный набор тайлов Чеченской Республики;
* локальные изображения;
* локальный API;
* локальную базу;
* автоматический запуск после включения компьютера.

Интернет нужен только для:

* обновления данных;
* отправки внешних заявок;
* удалённого администрирования;
* резервного копирования.

# 31. Критерии приёмки MVP

## Карта

* карта открывается;
* отображаются точки;
* карта масштабируется;
* точки можно выбрать;
* выбранная точка подсвечивается.

## Граф

* после выбора объекта появляется сеть;
* узлы можно нажимать;
* граф можно расширять;
* связи имеют тип;
* интерфейс не показывает более 40 узлов одновременно.

## Контент

* каждый опубликованный объект имеет источник;
* каждая опубликованная связь имеет источник;
* отсутствуют фиктивные факты;
* изображения имеют подписи и происхождение.

## Заявки

* пользователь может отправить заявку;
* заявка появляется в административной панели;
* модератор может её одобрить;
* после одобрения данные появляются в публичной части;
* отклонённая заявка не публикуется.

## Стабильность

* приложение работает не менее 30 минут без перезагрузки;
* после потери интернета основная музейная демонстрация продолжает работать;
* после бездействия приложение сбрасывает сеанс.

# 33. Риски

## Недостаток проверенного контента

Решение:

* ограничить MVP несколькими населёнными пунктами;
* использовать только подтверждённые материалы;
* не стремиться заполнить всю карту.

## Перегруженный граф

Решение:

* раскрывать по одному уровню;
* группировать узлы;
* ограничивать количество;
* добавлять кнопку «Показать ещё».

## Недостоверные пользовательские заявки

Решение:

* обязательная модерация;
* обязательное указание источника;
* уровни достоверности;
* журнал изменений.

## Зависимость от внешних карт

Решение:

* MapLibre;
* независимый поставщик тайлов;
* локальный набор карт для музея.

## Проблемы с интернетом

Решение:

* локальная база;
* локальные медиафайлы;
* локальная карта;
* offline-first архитектура.

## Слишком большой backend

Анти-паттерн:

```text
FastAPI + PostgreSQL + Redis + RabbitMQ + Celery +
Elasticsearch + Neo4j + Kubernetes
```

Для MVP это создаст слишком много точек отказа.

Минимально достаточная версия:

```text
React + MapLibre + Cytoscape
FastAPI + SQLite
```

# 36. Ключевое позиционирование

> «Паутина истории Чечни» — это интерактивная карта исторической памяти, где каждое село, человек, событие и достопримечательность становятся частью единой сети подтверждённых связей.
