# Техническое задание: «Паутина истории Чечни»

Статус: согласованная основа MVP

Формат: интерактивное веб-приложение для музейного сенсорного экрана и браузера

Заказчик и площадка: музей Чеченского государственного университета им. А. А. Кадырова

## 1. Проблема и результат

Исторические материалы о Чеченской Республике распределены между музеями, архивами,
публикациями и семейными коллекциями. Посетителю трудно увидеть связи между местами,
людьми, событиями, организациями и подтверждающими источниками.

Продукт должен позволить посетителю без обучения:

1. найти населённый пункт или исторический объект;
2. открыть связанные объекты как ограниченный граф;
3. перейти к карточке и подтверждающим источникам;
4. вернуться к карте или продолжить исследование;
5. предложить новый материал для ручной модерации.

Ценность экспоната — не справочник и не набор длинных статей, а визуальное исследование
проверенных исторических связей.

## 2. Известное, допущения и открытые вопросы

| Утверждение | Статус | Действие |
| --- | --- | --- |
| Главный сценарий работает на сенсорном музейном экране | Известно | Проектировать kiosk-first и проверять на 1920×1080 |
| Нужна публичная браузерная версия | Известно | Делать адаптивно от 360 px |
| Публичный контент обязан иметь подтверждающий источник | Известно | Запретить публикацию сущности и связи без источника |
| Основная демонстрация должна переживать потерю интернета | Известно | Локально хранить опубликованный read-only набор, карту и медиа |
| Offline-карта использует локальный versioned PMTiles artifact | Решение | Собирать из лицензированного OSM extract, сохранять attribution и provenance manifest |
| Заявки и модерация работают без интернета | Не требуется для MVP | Показывать понятное состояние «нужно подключение» |
| Чеченская локализация нужна целиком | Допущение | В MVP хранить RU и опциональный CE, объём уточнить по контенту |
| Конкретный список исторических материалов утверждён | Открыто | Владелец контента передаёт проверенный seed-набор |
| Права на фото и обработка контактов согласованы | Открыто | До публичного сбора данных нужна юридическая проверка |

## 3. Акторы и права

| Актор | Может | Не может |
| --- | --- | --- |
| Посетитель | смотреть опубликованные данные, искать, исследовать граф, отправлять заявку | видеть черновики, контакты и административные данные |
| Автор заявки | создать и изменить свой черновик, загрузить фото, отправить, посмотреть статус | изменять чужую заявку или публиковать материал |
| Модератор | видеть очередь, брать заявку в работу, запросить уточнение, отклонить, опубликовать | управлять аккаунтами и системной конфигурацией |
| Редактор | просматривать, создавать, исправлять, архивировать и экспортировать каталог с обязательными источниками | видеть контакты заявителей, назначать роли и управлять инфраструктурой |
| Администратор | использовать всю admin-панель, назначать роли и выполнять действия модератора/редактора | обходить аудит и контентные инварианты |

Неизвестная роль или разрешение всегда означает запрет. Скрытая кнопка на frontend не
заменяет backend-авторизацию.
Первый admin-аккаунт создаётся one-shot bootstrap-командой из production environment:
`ADMIN_BOOTSTRAP_EMAIL` и secret file, путь к которому задаёт
`ADMIN_BOOTSTRAP_PASSWORD_FILE`. Значения не хранятся в Git/Compose, не печатаются и не
используются для проверки каждого login. После bootstrap пароль хранится только как Argon2id
hash в PostgreSQL, а браузер авторизуется обычной серверной HttpOnly-сессией.

## 4. Объём MVP

### Обязательно

- стилизованная интерактивная карта Чечни на MapLibre GL JS;
- 8–10 проверенных населённых пунктов и не менее 50 подтверждённых связей;
- поиск по русским, чеченским, историческим и альтернативным названиям;
- фильтры по типу объекта, району и крупному историческому периоду;
- граф ближайших связей на Cytoscape.js;
- карточки сущностей, связей, источников и опубликованных фотографий;
- публичная заявка с несколькими фотографиями;
- очередь модерации и решения publish/reject/request revision;
- административная панель `/admin` с авторизацией, каталогом, модерацией и аудитом;
- просмотр, добавление, редактирование и архивация сущностей, связей и источников;
- bounded экспорт каталога в JSON или CSV без контактов, паролей и session data;
- автоматический сброс музейного сеанса;
- online production и локальная read-only музейная сборка;
- сценарные тесты критических потоков.

### После MVP

- игровой режим «Найди связь»;
- QR-коды и полноценная локализация;
- bulk-import, массовое редактирование и визуальный конструктор схем;
- синхронизация offline-заявок;
- аудио/видео, S3/MinIO и фоновые медиаконвейеры;
- рекомендательная система, AR, VR и мобильное приложение.

ИИ не создаёт исторические факты и не публикует материалы автоматически. Возможные
LLM-функции рассматриваются отдельно после появления проверяемого пользовательского
сценария, бюджета и правил обработки данных.

## 5. Основные сценарии

### 5.1. Исследование

Карта → поиск/выбор точки → карточка → ближайший граф → другой узел → источник → возврат.

На экране одновременно отображается не более 20 узлов по умолчанию и не более 40 после
явного раскрытия. Остальные связи представлены действием «Показать ещё» с количеством.

### 5.2. Публичная заявка

Выбор типа → связанный объект → описание и происхождение → контакты и согласие → фото →
проверка → отправка → неугадываемый tracking code.

Черновик принадлежит создавшему его браузерному сеансу. Знание UUID не даёт права менять
заявку. Tracking code криптографически случайный, ограничен rate limit и не раскрывает
контактные данные.

### 5.3. Модерация

Вход → очередь → заявка → проверка источников и медиа → решение → аудит → обновление
публичного каталога.

Публикация сущности, связей, источников, выбранных фото и статуса заявки выполняется одной
атомарной бизнес-операцией. Повторный запрос не создаёт дубликаты.

### 5.4. Административная панель

`/admin/login` → серверная сессия → `/admin` → каталог/модерация/экспорт.

Панель позволяет искать и фильтровать каталог, открывать полные карточки, создавать и
исправлять сущности, связи и источники, архивировать записи без физического удаления,
просматривать заявки и аудит, а также скачивать ограниченный экспорт JSON/CSV. Действия
доступны только согласно backend permissions и фиксируются в аудите.

### 5.5. Museum mode

После 30 секунд бездействия показывается предупреждение. Через 10 секунд без ответа
сбрасываются выбранные узлы, фильтры, поиск и временные данные посетителя. Опубликованный
контент остаётся доступен без интернета; online-only действия блокируются с объяснением.

## 6. Бизнес-правила

| ID | Правило | Владелец и отказ |
| --- | --- | --- |
| BR-001 | Опубликованная сущность имеет минимум один проверенный источник | catalog отклоняет публикацию с `source_required` |
| BR-002 | Опубликованная связь имеет минимум один проверенный источник | catalog отклоняет публикацию с `source_required` |
| BR-003 | Связь не соединяет сущность саму с собой | domain отклоняет с `self_relation_forbidden` |
| BR-004 | Публичное API возвращает только опубликованный контент | repository применяет обязательный published scope |
| BR-005 | Публиковать и отклонять может только разрешённая роль | moderation отклоняет с `forbidden` |
| BR-006 | Один moderation command создаёт не более одного результата | UoW/idempotency возвращает прежний результат или `conflict` |
| BR-007 | Фото не публикуется без происхождения и подписи | media/moderation отклоняет решение |
| BR-008 | Исторический факт не создаётся из непроверенного AI-ответа | процесс модерации требует источник |
| BR-009 | Изменение каталога через admin-панель требует роли editor или admin и создаёт audit record | catalog policy отклоняет с `forbidden` или откатывает операцию |
| BR-010 | Архивация скрывает запись из public API без физического удаления | catalog сохраняет историю и возвращает архивную запись только admin API |
| BR-011 | Экспорт не содержит credentials, sessions, контакты заявителей и внутренние storage keys | export service использует явный allowlist полей |

Статусы заявки:

```text
draft → pending → in_review → published
                    ├──────→ rejected
                    └──────→ needs_revision → pending
```

Промежуточный `approved` не используется: в MVP одобрение и публикация являются одной
атомарной операцией.

## 8. Данные и модули

Backend — модульный монолит:

- `catalog`: сущности, связи, источники, map/search/graph queries;
- `submissions`: публичный черновик, отправка и статус;
- `moderation`: очередь, решения и аудит;
- `media`: проверка, хранение, превью и публикация файлов;
- `auth`: аккаунты, серверные сессии, роли и разрешения.

Основные данные: `entities`, `entity_names`, `relations`, `sources`, привязки источников,
`media`, `submissions`, история статусов, moderation audit, accounts/sessions/roles.

Координаты хранятся в PostgreSQL 17 + PostGIS. SQLite и `.db` не используются. Миграции —
Alembic, одна бизнес-операция — один commit через Unit of Work.

Frontend-модули: `exploration`, `entities`, `submissions`, `moderation`, `auth`, `kiosk`.
Другие модули импортируют только публичный `index.ts` владельца.

## 9. Стек и инфраструктура

- Backend: Python 3.13, FastAPI, Pydantic v2, SQLAlchemy 2 async, Alembic, PostgreSQL/PostGIS.
- Frontend: React 19, TypeScript, Vite, MapLibre GL JS, Cytoscape.js, TanStack Query,
  React Hook Form, Zod.
- Zustand допускается только для доказанно сложного локального состояния карты/графа;
  серверные данные в нём не хранятся.
- Production: один canonical Docker Compose, Nginx, HTTPS, non-root containers.
- Медиа MVP: локальный Docker volume с adapter-интерфейсом. MinIO не требуется.
- Redis/RabbitMQ/Celery не используются новыми функциями без конкретной задачи.
- Admin UI публикуется тем же frontend bundle по `/admin`; отдельный сервис не создаётся.
- Bootstrap получает email из environment, а пароль через secret file path из environment;
  API/frontend containers не обязаны постоянно получать bootstrap secret.
- Offline map — локальный `chechnya.pmtiles`, собранный из OpenStreetMap extract; приложение
  показывает OSM attribution и поставляется с license/provenance manifest.

## 11. Приёмочные сценарии

### AC-001: исследование истории

```gherkin
Given опубликованный проверенный seed-набор
When посетитель находит место, открывает граф и выбирает связанную личность
Then он видит связь и минимум один подтверждающий источник без более 40 узлов на экране
```

### AC-002: защищённая заявка

```gherkin
Given посетитель создал черновик
When другой сеанс знает его UUID и пытается изменить или загрузить фото
Then backend возвращает not_found или forbidden и данные не меняются
```

### AC-003: атомарная публикация

```gherkin
Given заявка находится in_review и модератор имеет разрешение
When он публикует проверенную сущность, связь, источник и выбранные фото
Then всё появляется одной операцией, аудит сохранён, а повтор не создаёт дубликаты
```

### AC-004: отказ публикации

```gherkin
Given заявка не содержит подтверждающего источника
When модератор пытается её опубликовать
Then публикация отклоняется, публичный каталог не меняется и заявка остаётся доступна
```

### AC-005: offline museum mode

```gherkin
Given опубликованный набор загружен на музейный компьютер
When интернет пропадает во время исследования
Then карта, граф, карточки и источники продолжают работать, а online-only действия объяснимо недоступны
```

### AC-006: управление каталогом

```gherkin
Given editor вошёл через серверную сессию в /admin
When он создаёт сущность с источником, исправляет её и затем архивирует
Then каждое действие видно в аудите, public API показывает только актуальное опубликованное состояние
```

### AC-007: безопасный экспорт и bootstrap

```gherkin
Given первый admin создан one-shot bootstrap и вошёл в /admin
When он экспортирует каталог
Then ответ ограничен, содержит только разрешённые catalog fields и не содержит credentials, sessions или контакты заявителей
```

## 13. Решения и открытые вопросы

Принято: PostgreSQL/PostGIS, `/api/v1`, admin UI `/admin`, единый envelope, server-side
session, environment/secret-file bootstrap первого admin, RBAC, bounded catalog export,
локальное media-хранилище MVP, read-only offline, ручная публикация, вертикальная разработка.

До наполнения и публичного запуска нужно решить:

1. кто утверждает исторический seed-набор и каждую спорную связь;
2. точные тексты согласия, сроки хранения контактов и права на изображения;
3. реальное kiosk-оборудование для измерений FPS и времени загрузки;
4. обязательный объём чеченской локализации.
